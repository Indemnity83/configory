# AGENTS.md

Guidance for AI agents (and humans) working in this repository.

## Project

Configory is a small Gson-backed configuration library for Minecraft mods: typed config keys,
dot-notation paths (with a bare-key single-file mode → `config.json`), fluent validation,
dirty tracking, and lightweight bootstrap conventions. It is a single-module Java 21 Gradle
library. Source lives under `src/main/java/com/indemnity83/configory/`.

## Build, test, format

Use the Gradle wrapper. The daemon is pinned to JDK 21 via `gradle/gradle-daemon-jvm.properties`
(the Palantir formatter can't run on newer JDKs), so `./gradlew` works regardless of your
default `java`.

- `./gradlew test` — run the JUnit 5 suite.
- `./gradlew lint` — check formatting (Spotless, `palantirJavaFormat`, 4-space).
- `./gradlew fix` — apply formatting. Run this before committing.
- `./gradlew test jacocoTestReport` — tests plus coverage (what CI runs).

Tests use an in-memory `ConfigStorage` double (`InMemoryConfigStorage`) — do not touch the real
filesystem in unit tests.

## Git workflow

This repo uses [Git Town](https://www.git-town.com/). Prefer `git town` commands for all git
work; fall back to plain `git` only for operations Git Town doesn't cover.

- `git town hack <branch>` — start a new feature branch off `main`.
- `git town append <branch>` — stack a new branch on top of the current one; use this to build
  a chain of dependent branches/PRs instead of piling unrelated work into one branch.
- `git town sync` — pull `main` and update the current branch (and the rest of the stack).
- `git town propose` — open a pull request for the current branch.
- `git town ship` — merge a reviewed, green branch and clean it up.

Branch with `git town hack` and propose a PR rather than committing to `main` directly. For
work that depends on an unmerged branch, `git town append` to keep it as a stacked chain.

## Commits & pull requests

- **No agent co-author trailers.** Do not add `Co-Authored-By: <agent>` or any
  "generated by" footer to commits or PR bodies — it's considered noise.
- **Conventional commits**, no scopes. Allowed types: `feat`, `fix`, `deprecate`, `remove`,
  `security`, `perf`, `docs`, `refactor`, `test`, `build`, `ci`, `chore`, `revert`.
- **PR title subject starts uppercase**, e.g. `feat: Add long range constraints`,
  `fix: Prevent a stuck config value`. Enforced by the `Check PR` workflow.
- `main` is protected: PRs require the `lint`, `test`, and `lint (pr)` checks to pass and to be
  up to date.

## Pull request body

Write the PR body like **release notes** for users and maintainers — focus on *what* changed
and *why* it matters, not low-level implementation details. Keep it concise and scannable, use
markdown headings and bullets, don't invent changes that aren't in the diff/commits, and avoid
phrases like "this PR" / "this branch".

Structure:

- **Summary** — a short paragraph describing the purpose of the change.
- **Changes** — bullet points of the major changes, grouped where it helps.
- **Notes** (optional) — compatibility notes, migrations, or contributor info. **When the PR
  resolves an issue, add a `Fixes #<n>` or `Closes #<n>` line here** so the issue auto-closes
  on merge.

## CI & automation

- `Check Code` (`lint` + `test`) runs on push/PR to `main`.
- `Check PR` lints the PR title.
- Dependabot (gradle + github-actions, weekly). Minor/patch updates auto-merge once green;
  major updates wait for manual review.
- Release automation is release-please (`Prepare Release`); the published version comes from
  the `MOD_VERSION` env var at build time (`dev` locally).

## Code comments

Avoid comments. Reserve them for genuinely cryptic or unintuitive code that can't be made
clearer any other way. If you feel the urge to add a comment explaining what some code does,
treat it as a signal that the code itself needs work: extract a well-named method, rename a
variable, or restructure so the code is self-documenting. A comment that restates what the code
already says is noise — delete it and improve the code instead.

Never write comments that describe history or changes — why a line was edited, what it used to
be, or that it changed in response to some request (e.g. "changed to fix…", "now uses…",
"previously…"). Code describes the present; history lives in git. Such comments are always noise
and never help a future reader.

Tests are looser. A test's setup, action, or expected outcome isn't always obvious from the
code, so explanatory comments there are welcome and encouraged where they aid understanding. The
history rule still holds even in tests — don't describe how the code changed unless that change
is exactly what the test is asserting.

## Public API documentation

Documentation is not a comment, and the rules above do not apply to it. This is a library, so
its public API is a product surface that should be documented. **Every public type carries a
class-level Javadoc block** describing what it is and how it's used, and public methods and
fields are documented where their contract isn't obvious from the signature. Write Javadoc for
the consumer reading it from an IDE or generated docs — describe behavior, parameters, return
values, and thrown exceptions, not implementation.

## Conventions

- Match the style and idiom of surrounding code.
- `config` is the reserved default file name; a bare key (no dot) lands in `config.json`, and
  `config.<key>` is an accepted alias of the bare form.
